# Spec-Driven Development (SDD) Approaches

| Approach / Tool | Primary Focus | Mechanism | Best For |
| :--- | :--- | :--- | :--- |
| **GitHub Spec Kit** | Lightweight scaffolding | CLI-tool to define structure | Rapid prototypes, common patterns |
| **Amazon Kiro** | Agentic requirement-to-task | IDE-embedded agent loop | End-to-end SDD, enterprise workflows |
| **Tessl** | Specification as Source Code | Model-platform integration | Radical AI-native, specification-first |
| **OpenSpec** | Framework for SDD | Standardizing spec-driven loops | Teams building custom coding agents |

## Overview
Spec-Driven Development (SDD) has emerged as the structured answer to the "productivity paradox" in AI coding. While current tools prioritize different points in the development lifecycle (from scaffolding to full IDE-agent integration), they all converge on the necessity of explicit specification discipline over implicit AI "vibe coding."

## Dimensions of Comparison
1. **Automation Level**: Ranges from developer-run CLI tools (Spec Kit) to autonomous agentic IDEs (Kiro, Tessl).
2. **Standardization**: Some frameworks (OpenSpec) prioritize interoperability, while IDE-integrated tools (Kiro) provide a prescriptive environment.
3. **Integration**: Methods vary between external tool chains (Spec Kit) and deep model-coupling (Tessl).

## Synthesis
The choice of SDD implementation depends on the desired level of agent autonomy vs. developer control. Organizations requiring rapid, modular scaffolding might favor Spec Kit; those seeking unified, agent-managed IDE environments are better suited for Kiro or Tessl.
