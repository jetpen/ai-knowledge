# Hermes Agent: Knowledge Architecture

## Overview
Hermes Agent uses a memory architecture loosely based on cognitive science taxonomies, separating knowledge into four distinct persistent stores. This separation is enforced at the system prompt level rather than just theoretically.

## The Four Knowledge Types
1. **Declarative Memory** (What's true): Bounded plaintext files:
    * `MEMORY.md` (2,200 chars limit): Environment and facts.
    * `USER.md` (1,375 chars limit): Personal user data.
    * Persistent across sessions; updated by the agent.
2. **Procedural Memory** (How to do things): `SKILL.md` (unbounded, agent-created). Created automatically after complex multi-step tasks (5+ tool calls).
3. **Episodic Memory** (What happened): Passively logged in SQLite + FTS5. Retrieved only on demand.
4. **Identity**: Assembled from layers (`SOUL.md`, personality presets, platform hints, Honcho AI peer representation).

## Memory Management Principles
*   **Routing Logic**: A deterministic decision tree (`_build_system_prompt()`) determines where information is stored:
    * User-related facts -> `USER.md`
    * Reusable procedures -> `SKILL.md`
    * Environment facts -> `MEMORY.md`
    * Task-specific state -> Stored in standard `SessionDB` (not persisted in declarative memory).
*   **The Bounded Constraint**: Bounded storage forces the agent to curate information, preventing "memory hoarding" and retrieval noise.
*   **Frozen Snapshot Pattern**: Disk memory loads once at session start and is injected as a static snapshot. This keeps the System Prompt cache stable across turns, saving costs by avoiding repetitive context reprocessing.
*   **Consolidation Cycles**: Uses `nudge_interval` (triggered turn count) to prompt the agent to evaluate if information is worth persisting.

## Security
*   All memory writes are scanned by `_scan_memory_content()` for injection, exfiltration, or backdoor patterns before writing.
*   Writes are atomic via `os.replace() + fsync()`.
