# Harness Engineering

## Overview
Nav Toor argues that "Harness Engineering" is the critical skill of 2026. Rather than relying on model improvements, elite engineering teams achieve massive performance gains by optimizing the "harness"—the system of rules, tools, skills, memory, and feedback loops surrounding the LLM.

## Core Philosophy
"Anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent never makes that mistake again." — Mitchell Hashimoto.
Do not pray for better models; fix the system around the model.

## The Five Levers of a Coding Agent Harness
1. **System Prompt (e.g., `CLAUDE.md`)**: Injected into context at session start. Must be concise (<60 lines), specific, and human-written. Avoid AI-generated instruction files.
2. **Skills (Progressive Disclosure)**: Modular instruction files loaded only when needed. Prevents context bloat by keeping irrelevant instructions out of the active window.
3. **MCP Servers**: Tools and external service connectors (e.g., Linear, Sentry). Avoid "tool thrash" by starting with a minimal set.
4. **Sub-agents (Context Firewalls)**: Segregating tasks into isolated agent sessions prevents context pollution and keeps logic focused.
5. **Feedback Loops**: Mechanisms to catch and correct mistakes (e.g., automated test failure catchers) before they impact production.

## Key Takeaways
*   Harness design is responsible for performance leaps (e.g., 42% to 78% on coding benchmarks with the same model).
*   Progressive disclosure of knowledge (Skills) is superior to monolithic system prompts.
*   The developer of 2026 is less a coder and more a "Harness Engineer."
